# FOSS Adoption: The "Secure AND Tested" Gate

Companion reference for the senior-engineering-partner skill.

Free and open-source software is the default, not a concession — but "free" buys you a maintainer's goodwill, not their accountability. Every dependency you `import` is code you now ship, code an attacker can target, and code that can rot under you. This doc is the **decision checklist for whether to adopt at all**; the *mechanics* of locking, auditing, and supply-chain vetting (lockfiles, `npm audit`, install-script hardening) live in `references/package-managers.md`, and the pin/lock discipline is owned by SKILL.md "DEPENDENCY MANAGEMENT". Adopt nothing that fails the gate below.

---

### The gate in one line

A FOSS dependency may be adopted only when it is both **secure** (license-clean, CVE-clean, healthily maintained, small attack surface) and **tested** (pinned, scanned in CI, advisory-subscribed, and wrapped in a contract test). Pass both halves or do not adopt.

## Pre-adoption checklist (the "secure" half)

- **License — OSI-approved AND compatible with *your* use.** Confirm the license is on the OSI-approved list and that it fits your distribution model, then read the actual `LICENSE` file — don't trust the SPDX tag alone. **Permissive** (MIT, Apache-2.0, BSD-2/3-Clause) is the safe default for a commercial/SaaS product; Apache-2.0 additionally carries an explicit patent grant. **Strong copyleft** (GPL-3.0, and especially **AGPL-3.0**) is the trap: AGPL's Section 13 ("Remote Network Interaction") obligates whoever offers a *modified* version to network users to provide those users the Corresponding Source — i.e. exactly a SaaS backend — so an AGPL dependency you modify and run server-side can obligate you to release source. Whether the clause reaches *your* code depends on whether you've modified the covered work and how it's combined — that is **legal judgment, not a checkbox** — surface it to counsel and verify the precise license text against the upstream `LICENSE`. A repo with **no license is not "permissive," it is legally unusable** — default copyright reserves all rights; reject it.
- **Maintenance health — is anyone home?** Look for recent commit and release cadence (a tag in the last 6–12 months, not a single stale commit from years ago), more than one maintainer (**bus factor > 1**), a healthy open-vs-closed issue ratio, *responsiveness to security reports*, and a real `SECURITY.md` with a disclosure channel. Quantify it instead of eyeballing: run the **OpenSSF Scorecard** (`ossf/scorecard`) against the repo and read **deps.dev** (Google Open Source Insights) for the package's health and dependency signals. Confirm each tool's *current* invocation and the exact set of checks against its own docs before quoting a number — tool surfaces and check names drift between releases (verify against current docs). **Scorecard is bidirectional** — the same tool run on your *own* repo is a posture self-assessment + public badge (`compliance.md` → *OpenSSF Scorecard — run it on your OWN repo*).
- **Security posture — clean and provable.** Pull the known-vulnerability history before you commit: **osv.dev** is the cross-ecosystem source of record, queryable via **osv-scanner** against a lockfile; per-ecosystem, **pip-audit** (Python) and `npm audit` (Node) hit overlapping advisory data, and **trivy** scans filesystems and images. Verify each tool's current subcommand/flags against its docs (e.g. osv-scanner's `scan source --lockfile=…` and trivy's `fs <path>` forms have both shifted across major versions — verify against current docs). Beyond CVEs, prefer projects with **signed releases / build provenance** (e.g. Sigstore-backed attestations, or npm package provenance) and maintainers with **2FA enforced** on the registry account — a hijacked unprotected maintainer account is the classic supply-chain entry point.
- **Footprint — attack surface, not just disk size.** The transitive dependency count *is* your attack surface: a package that pulls 200 transitive deps means 200 maintainer accounts that can compromise you. Flag any dependency with **native build steps or install lifecycle scripts** (`postinstall`, build hooks, `setup.py` running arbitrary code) — that is a code-execution vector at install time, not merely a convenience; harden per `references/package-managers.md`. Weigh install size and cold-start/bundle impact, but rank *provenance of every transitive node* above raw megabytes.
- **Fit — do you actually need it?** A dependency you can replace with a dozen lines of standard library is a liability you took on for nothing (the `left-pad`/`is-even` class of mistake). Conversely, **popularity is not quality** — download count measures inertia, not safety — and **abandonware is a live risk** even at high star counts. Adopt when the problem is genuinely hard (crypto, parsing, protocol handling — where rolling your own is *more* dangerous) and the project is healthy; otherwise vendor a small, audited snippet.

## After adoption (the "tested" half)

This is the half teams skip, and it is what makes "tested" real rather than aspirational.

- **Pin and lock — exactly.** Pin to an exact version and commit the lockfile so every install is reproducible; this is owned by SKILL.md "DEPENDENCY MANAGEMENT" and `references/package-managers.md`. Floating ranges (`^`, `~`, `>=`) re-open the supply-chain door on every fresh resolve.
- **Add it to the continuous gates.** The same scanners you ran once (osv-scanner / pip-audit / `npm audit` / trivy) belong in **CI on every PR and on a schedule**, failing the build on new advisories — a clean scan at adoption time is a snapshot, not a guarantee. Wire it into the project's existing test job (verify the runner's current syntax against its docs).
- **Subscribe to advisories.** Turn on automated update PRs — **Dependabot** (GitHub-native, sourced from the GitHub Advisory Database) or **Renovate** (cross-forge, which reads both OSV and GitHub advisory data). Verify current config-file keys against each tool's docs before writing one. Critical caveat (re-confirm the current threat landscape): bot-authored update PRs carry *implicit trust* and have been used as a malware-delivery vector — **do not auto-merge**; require review, and prefer a release-age cooldown (Renovate exposes a `minimumReleaseAge` setting; verify the current key name and accepted values against Renovate docs) so a freshly-published malicious version sits before it can reach you.
- **Write a THIN contract test.** Wrap the dependency in one small integration test that exercises *the exact behavior you depend on* — the function signature, the field you read, the error you catch. This is the point of the whole exercise: when a "patch" bump silently changes a return shape or drops a field, the contract test goes **red in CI instead of red in production**. Test the seam you actually rely on, not the library's whole API.

## Tie the rigor to phase (Rigor Ladder)

Match the depth of this gate to the tier from SKILL.md's "PROJECT PHASE & RIGOR LADDER" — full provenance vetting on a throwaway script is theater; skipping it on production is negligence.

- **Tier 0–1 (throwaway / personal tooling / early MVP):** a quick glance suffices — license is OSI-approved and permissive, no open critical advisories (one `osv-scanner` / `pip-audit` run), and the project isn't visibly abandoned. Pin it; move on.
- **Tier 2 (production / multi-tenant / anything customers touch):** the **full checklist is mandatory** — license reviewed for copyleft/AGPL reach, Scorecard + deps.dev health read, advisory scan clean, **signed-release / provenance** confirmed, transitive footprint and install-script audit done, contract test written, CI gates and advisory subscription wired. No exceptions for "it's just a small helper."

> **Environment-specific examples** — for *your* concrete stack, the AGPL-network-clause exposure applies directly to a modified AGPL component running in the example SaaS Cloud Run backend, and CI advisory gates run in GitHub Actions per the org's PR-flow rules. Keep those specifics in `references/my-environment.md` rather than baking them in here; this checklist is stack-agnostic by design.
