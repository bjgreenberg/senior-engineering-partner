# Local & Agentic AI Tooling

Companion reference for the senior-engineering-partner skill.

An agentic coding assistant is a junior engineer with commit access and a terminal; a self-hosted LLM is a network service with no front door. Both are *defaults-hostile*: the agent will auto-accept and over-reach unless you scope it, and the local inference server ships with a confidentiality and exposure posture you must verify and tighten yourself. The disciplines below are the team-flow and least-privilege rules this skill already mandates, applied to the case where the developer is a model and the dependency is a binary you run on your own box. Treat every version, UI, default, port, and config-key claim here as something to confirm against current product docs before you rely on it — that is the skill's anti-hallucination standard, applied to its own subject.

## Part A — Agentic coding assistants (Claude Code, Codex, Antigravity)

**The model is the junior engineer; you are still the reviewer of record.** The tool is interchangeable — the same discipline applies whether the runner is Claude Code, OpenAI Codex, or Google Antigravity. Autonomy raises the blast radius of a bad diff, a leaked secret, or an unguarded `rm`; it does not change who is accountable for what lands.

- **Review every diff — no blind auto-accept.** An agent edit is a human PR (cross-ref `references/dev-environments.md`, where the Antigravity section makes this explicit). Read the change, not just the summary; an "agent diff = human PR" posture means the same scrutiny you'd give a teammate's branch. Auto-accept-all turns a confident hallucination into a committed fact.
- **Least-privilege scope — one project, one worktree.** Run the agent rooted in a single repo or `git worktree`, never `$HOME` or a parent dir that contains unrelated repos. Keep it away from `~/.ssh`, the SSH-agent / 1Password agent socket, browser profiles, and credential stores — anything in the open files, terminal scrollback, or working dir can enter model context and leave the machine. Narrow the cwd *before* you start; you can't un-send context.
- **Secrets out of model context — reference, don't resolve.** Never paste a resolved secret (an `op read` result, a `.env`, a service-role key, SA JSON) where the agent can read it. Reference secrets by 1Password path and let the *runtime* resolve them at exec time, outside the agent's view (this is the skill's Secrets Management discipline; environment specifics — vaults, which keys — live in `references/my-environment.md`). A secret that never enters context can't be exfiltrated by an injected instruction or logged into a transcript.
- **Keep the confirmation/allowlist posture ON — never blanket-allow destructive commands.** Destructive or irreversible actions stay behind an explicit human gate: `rm`/`unlink`, `git push --force`, `DROP`/`DELETE`/`TRUNCATE` against a real database, cloud mutations (`gcloud`/`gsutil`/`aws`/`terraform apply`), dependency removals, credential rotation. Allowlist *read-mostly* commands if you must; never widen the allowlist to cover the irreversible set just to stop the prompts. (Worked example of the sharp edges — a deliberately-public bucket an agent must not "tidy" private, RLS policies that are tenant boundaries — is in `references/my-environment.md`.)
- **Agent output rides branch → PR → required CI, exactly like a human's.** The agent commits to a feature branch and lands via PR with every security/integration check marked *required* — not direct-to-`main`, not blind self-merge. This is the concurrency override in full: read `references/multi-agent-coordination.md` (one worktree/branch/task per agent, `git pull --rebase` before push, stage by explicit path) and `references/github-teams.md` (CODEOWNERS, a human reviews every agent PR). The merge gate is mechanical, not trust-based.
- **Audit what it executed.** Log the commands the agent ran and route those logs per SKILL.md *Log location, rotation & retention* — `~/Library/Logs/<tool>.log`, rotated, `chmod 600`, never committed (runtime output is not a dotfile). You want an after-the-fact record of what touched the filesystem and the network, separate from the chat transcript.

### Tool-specific posture (verify each against current docs — do not assume a flag exists)

- **Claude Code** — gate execution through its permission/approval model: a settings file with explicit allow/deny lists and hooks that fire on tool use. *Verify the exact mechanism — file name, the allow/deny/ask keys, and hook event names — against current Claude Code documentation before relying on a specific one; the durable rule is "default-confirm, allowlist read-mostly, deny the irreversible set," not any one config key.*
- **OpenAI Codex** — separates *sandbox* (what a command can technically touch — e.g. a read-only vs. workspace-write vs. full-access boundary) from *approval policy* (when it must ask). Keep it sandboxed to the workspace with approvals on, so it asks before it leaves the sandbox, reaches the network, or runs outside a trusted set; treat the full-access-plus-never-approve combination as off-limits for any repo with real credentials or a live DB. *Verify the current sandbox/approval value names, the config file and keys, and exact defaults against OpenAI's current Codex docs — do not hardcode a value from memory.*
- **Google Antigravity** — gates terminal commands behind a per-command confirmation prompt by default (not in a blanket-allow mode), driven by configurable Allow / Deny lists; leave that posture on and do not switch it to run every command without asking. Full editor-hygiene detail (Workspace Trust, the diff-equals-PR table, where its context goes) lives in `references/dev-environments.md` — point there, don't duplicate. *Verify the current mode names, the permission model, and defaults against current Google Antigravity docs; it is new and changes between releases.*

- **Unattended agent commits follow the per-invocation signing exemption.** A scheduled or bot-driven agent commit cannot interactively sign (1Password may be locked when it fires), so it uses `git -c commit.gpgsign=false commit …` per-invocation — never a per-machine config change. Include the flag from day one in any auto-committing agent (precedent, rationale, and the house Git standards behind it are in `references/my-environment.md`).

## Part B — Self-hosted / local LLM inference (Ollama, Open WebUI)

**Running inference locally is a confidentiality control — and a network service you now own.** Data locality is the legitimate reason to self-host: sensitive, PII, or tenant data stays off third-party APIs, which directly supports the no-train / minimal-retention posture in `references/data-protection.md`. But the moment you stand up the inference server you've added a service with no front door, an arbitrary-code model loader, and an unbounded appetite for RAM/VRAM. Local does not mean trusted.

### Network exposure — the headline risk

- **The inference API ships with no built-in authentication — keep it on loopback.** Ollama's server listens on a loopback host/port by default and has *no* authentication: anyone who can reach the port can list models, run prompts, pull/delete models, and burn GPU. *It is widely documented as binding `127.0.0.1:11434` by default — confirm the exact bind address and port against current Ollama docs and your installed version before stating them.* Do not rebind the host to all-interfaces (`0.0.0.0`) or otherwise expose it to the LAN or internet — mass-exposed-Ollama incidents are real, not hypothetical.
- **For remote access, front it with auth — never raw.** If you genuinely need the API off-box, put it behind an authenticating reverse proxy (TLS + a credential), an SSH tunnel, or a VPN. The proxy supplies the authentication the engine lacks. *The bind address is controlled by the `OLLAMA_HOST` environment variable (verify against current Ollama docs); the durable rule is "the engine has no auth, so something in front of it must."*
- **Open WebUI: the first account created becomes admin — never expose first-run.** On a fresh instance the first registered account becomes the administrator, and signup auto-closes after it; so an Open WebUI reachable on the public internet during first-run is an account-takeover risk — whoever hits it first becomes admin. Create the admin on loopback, set a strong unique password (injected, never hardcoded in a compose file), serve only over HTTPS via a TLS-terminating proxy, and do not re-open registration on an instance reachable beyond a trusted network. *Verify the current admin/signup model and the admin-password env var against Open WebUI's current docs before relying on a specific name or default.*

### Model supply chain

- **Pull from trusted registries; treat arbitrary weights as untrusted code.** A downloaded checkpoint is an executable until proven otherwise. Pull models from a registry you trust, and treat random weights, checkpoints, or GGUF files from the internet the way you'd treat a random binary.
- **Prefer safetensors over pickle-based formats.** Pickle-based serialization (PyTorch `.bin`/`.pt`/`.ckpt`) can execute arbitrary code *on load* via deserialization — loading the file is enough to be compromised. The safetensors format stores raw tensor data with no embedded code path and is the safer default. *Verify which on-disk format a given loader actually consumes before assuming it's safe — and note that "not pickle" is not "trusted": a GGUF file can carry a Jinja2 chat template that, in a vulnerable loader, becomes server-side template injection and code execution (the CVE-2024-34359 "Llama Drama" class — verify whether your runtime/version is affected against current advisories rather than assuming immunity).*

### Isolation & resource limits

- **Run it containerized with CPU/RAM/VRAM caps.** A local model can exhaust the box — a malicious GGUF or an oversized request can OOM or pin the GPU. Run the engine and the WebUI in containers with explicit CPU/memory limits, and bound VRAM where the runtime supports it, so one model can't take down the host. (Container hardening — non-root, pinned digests, scanned images — is `references/containers-and-orchestration.md`.)
- **No secret access from the inference stack.** The local LLM and its WebUI have no business reading `~/.ssh`, the SSH-agent / 1Password socket, cloud credentials, or a `.env`. Run them as a least-privilege user with a minimal mounted filesystem; an inference server is a text-in/text-out service, not something that needs your keychain.
- **The chat-history store is sensitive data — protect it, don't commit it.** Open WebUI persists conversations (and whatever PII/tenant content you fed it) in its datastore. Treat that store as sensitive: restrict file permissions, back it up like the confidential data it is, and never commit it or its volume to a repo (same class as logs — runtime output, not a dotfile).

### Output is still untrusted

- **Local inference does not launder the output.** The indirect-prompt-injection and output-validation rules from `references/secure-data-processing.md` apply unchanged: content fed to a local model is data, never instructions, and the model's output is validated against a schema and allowlisted before it drives any action or gets persisted as an artifact. Self-hosting changes *where the data sits*, not *whether the output can be trusted* — a poisoned document injects a local model exactly as it would a hosted one.
