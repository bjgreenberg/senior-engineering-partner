# UI Design & Accessibility

Companion reference for the senior-engineering-partner skill.

Every web or GUI front-end you ship is a deliverable with three co-equal, non-negotiable properties: it is **beautiful by default**, it ships **light AND dark mode**, and it meets **WCAG 2.2 Level AA**. "It works" is not the bar — an inaccessible UI is a defect and a legal-exposure surface, and an ugly one is a credibility leak. This doc is the home for the standards that SKILL.md's "Web & GUI front-end" section points to; it is stack-agnostic, with any environment-specific facts deferred to `references/my-environment.md`. It complements — does not replace — that section's responsive floor (mobile-first, Flexbox/Grid, breakpoints 480/768/1024/1280px).

---

## 1. Design-quality baseline (beautiful by default)

Beauty here is not taste-by-vibes; it is a small number of disciplines applied consistently. The mechanism is the same in every framework: **a token layer is the single source of truth, and components consume tokens — never literals.**

- **Design tokens first.** Define color, a type scale, a spacing scale, radius, shadow/elevation, and motion (duration + easing) as named tokens. Components reference `--color-surface`, `--space-4`, `--radius-md` — never `#1a1a1a`, `16px`, `6px` inline. Common carriers: CSS custom properties (`:root { --space-4: 1rem; }`), a Tailwind theme (configured via `theme.extend`, or a token layer in newer Tailwind — verify the exact directive/version against current Tailwind docs), or a Style Dictionary build. The carrier is interchangeable; the discipline is not.
- **A real typographic scale.** Pick a modular scale (e.g. a 1.2–1.25 ratio) instead of ad-hoc sizes, and cap body copy line length around 60–75 characters. Set line-height by role (tighter for headings, ~1.5 for body). Two font families maximum; let weight and size carry hierarchy.
- **A spacing rhythm.** One base unit (commonly 4px or 8px) and a scale derived from it. Every margin, padding, and gap is a scale step. Consistent rhythm is most of what separates "designed" from "assembled."
- **Restrained, consistent color.** One or two brand hues plus a neutral ramp; reserve saturated accent color for a single job (primary action, focus, alert). Derive semantic roles from the ramp rather than hand-picking per component.
- **Hierarchy and whitespace.** Establish a clear primary action per view; use size, weight, and spacing — not boxes and borders — to rank content. Whitespace is a feature; let elements breathe.
- **DESIGNED empty, loading, and error states.** These are first-class screens, not afterthoughts. Empty states explain and offer the next action; loading uses skeletons or deliberate spinners (avoid layout shift); errors are specific, recoverable, and human. A view is not "done" until all four states (default/empty/loading/error) are designed.
- **Restrained micro-interaction.** Motion clarifies state change and directs attention — hover/press feedback, enter/exit transitions. Keep it short (a low-hundreds-of-milliseconds range is typical; tune to feel) and purposeful. No motion for motion's sake, and everything degrades under reduced-motion (see §3).

## 2. Light & dark mode (MANDATORY for GUIs)

Dark mode is a baseline expectation, not a stretch goal. The only sane way to ship both is **semantic tokens with a theme-swapped value layer.**

- **Semantic tokens, never raw hex in components.** Components reference *roles* — `--color-bg`, `--color-surface`, `--color-text`, `--color-text-muted`, `--color-border`, `--color-primary`, `--color-danger`. The hex values live one layer down and are swapped per theme. A component should be theme-agnostic by construction.
- **Honor `prefers-color-scheme` AND a persisted manual toggle.** Default to the OS preference via the media query, but give users an explicit override that persists (e.g. `localStorage` + a `data-theme` attribute on the root, with the media query as fallback when no override is set). Set the CSS `color-scheme` property (e.g. `color-scheme: light dark`) so native form controls and scrollbars theme correctly. Avoid the first-paint flash by applying the stored theme before the app renders (an inline pre-render script is the usual approach — verify specifics for your framework).
- **Verify contrast in BOTH themes.** A palette that passes in light routinely fails in dark (and vice versa). Dark surfaces often need slightly desaturated/lifted text and elevation-by-lightness rather than heavy shadow. Every contrast assertion in §3 must hold in *each* theme — test both, not one.

## 3. WCAG 2.2 Level AA (MANDATORY)

Target conformance is **WCAG 2.2, Level AA.** The success-criteria numbers, names, levels, and thresholds below are stated from the WCAG 2.2 W3C Recommendation; **re-verify every SC number, name, and level against the live spec at `https://www.w3.org/TR/WCAG22/` before citing them in a compliance context** — do not quote SC numbers from memory.

- **Semantic HTML first; ARIA only to fill gaps.** Use `button`, `a`, `nav`, `main`, `label`, `fieldset`, native form controls. Reach for ARIA only where no native element exists — and remember the first rule of ARIA is *don't use ARIA if HTML can do it*. Bad ARIA is worse than none.
- **Contrast ratios.** Body text **4.5:1** (SC 1.4.3 Contrast (Minimum), AA); large text **3:1**, where the spec defines large text as roughly 18pt / 24px, or 14pt / ~18.66px bold and larger (confirm the exact threshold against the spec). Non-text UI components and graphical objects are **3:1** (SC 1.4.11 Non-text Contrast, AA — this covers input borders, meaningful icons, focus indicators).
- **Full keyboard operability + visible focus + logical order.** Everything actionable works from the keyboard with no traps. A visible focus indicator is required (SC 2.4.7 Focus Visible, AA) — never `outline: none` without an equal-or-better replacement; build the focus style from a token so it themes. WCAG 2.2 added SC 2.4.11 Focus Not Obscured (Minimum) (AA): a focused element must not be *entirely* hidden behind sticky headers/footers or similar author content. Note SC 2.4.13 Focus Appearance is **AAA**, not AA — treat its size/contrast guidance (a focus area at least as large as a 2px-thick perimeter, contrasting 3:1 with adjacent colors — verify against the spec) as a quality target, not a required floor.
- **Accessible names, roles, labels.** Every control has an accessible name (a visible `label`, or `aria-label`/`aria-labelledby` where a visible label is genuinely absent). Icon-only buttons always carry a name. Decorative images use empty `alt=""`; meaningful images get descriptive `alt`.
- **Honor `prefers-reduced-motion`.** Wrap non-essential animation/transition in `@media (prefers-reduced-motion: reduce)` and disable or substitute a cross-fade. Be precise about provenance: the related normative criterion, SC 2.3.3 Animation from Interactions, is **AAA** — supporting `prefers-reduced-motion` is mandated by *this house standard* as best practice, **not** by a WCAG 2.2 **AA** criterion. Don't claim AA forces it.
- **Target size.** This skill's stricter house rule is **44×44 CSS px** minimum touch targets (matching the figure in the AAA SC 2.5.5 Target Size (Enhanced) and common platform HIGs). WCAG 2.2's own **AA** floor, SC 2.5.8 Target Size (Minimum), is smaller — **24×24 CSS px**, with documented spacing/inline/equivalent exceptions. Build to 44; 24 is the conformance minimum, not the goal.
- **Identified, announced form errors.** Errors are identified in text (SC 3.3.1 Error Identification, A), tied to their field via `aria-describedby`, never signaled by color alone, and surfaced to assistive tech (e.g. an `aria-live` region or moving focus to an error summary). Provide labels/instructions before entry (SC 3.3.2 Labels or Instructions, A).
- **No color-only signaling** (SC 1.4.1 Use of Color, A). Pair every color cue with text, an icon, or a pattern — required/error/success/selected states must survive grayscale.
- **Skip links and landmarks.** Provide a "skip to main content" link as the first focusable element; structure pages with landmark regions and a single `h1` plus a sane heading outline.
- **Live regions for async updates.** Toasts, validation, and content that changes without a navigation use `aria-live` (`polite` for status, `assertive` only for genuinely urgent) or an appropriate `role="status"`/`role="alert"` so screen-reader users aren't left behind.

### Testing gate

- **Automated, in CI.** Wire an axe-based check (via your runner — e.g. an axe integration for Playwright, or jest-axe), Lighthouse's accessibility category, and/or `pa11y` into the pipeline and fail the build on regressions. Verify current package names and flags against their own docs.
- **Manual smoke pass, every PR with UI.** Tab through the whole flow keyboard-only (focus visible, order logical, no traps), then run one real screen reader (VoiceOver on macOS, NVDA on Windows) over the changed surface.
- **Honest caveat.** Automated tooling catches only a *fraction* of issues — commonly cited as roughly a third of WCAG success criteria by coverage, though vendor studies measuring issue *volume* report higher numbers (figures vary widely by source and methodology — verify before quoting a specific percentage). Passing the automated checks with zero violations is necessary, never sufficient. The keyboard + screen-reader pass is the real gate.

## 4. Claude Design (or better)

Anthropic Labs ships **Claude Design**, a front-end-focused visual/prototyping tool (research preview at the time of writing). **It is fast-moving and changes between releases — verify every specific below against current product docs before relying on it; treat anything not in the official announcement as unconfirmed.** Confirmed from Anthropic's launch announcement: it produces real runnable front-end artifacts (the announcement cites export to standalone HTML among other formats), during onboarding it **reads your codebase and design files to build a design system** it then applies to subsequent work, and when a design is ready it **packages a handoff bundle you pass to Claude Code** to build from.

- **Treat newer feature claims as version-gated.** A later overhaul reported by third-party outlets adds **design-system import from GitHub repositories**, a terminal `/design` handoff command, and a bidirectional design↔code round-trip — these are **not in the original launch announcement** and are corroborated only by secondary reporting, so confirm them against live product docs before depending on any of it. A `/design-sync` command specifically is unverified in the sources reviewed here — do not assume it exists. This skill's environment also exposes a `DesignSync` tool; **verify its current behavior directly rather than assuming it maps to any documented command.**
- **Feed it your tokens so output is on-brand and theme-correct.** Point it at your design-system/token source (§1) and your light/dark semantic tokens (§2) up front, so generated UI inherits your scale, color roles, and both themes instead of inventing a parallel system you then have to reconcile.
- **It is front-end only — never expose backend secrets or tenant data.** Its codebase read is *model context*; treat it exactly like any other agentic tool per `references/local-and-agentic-ai-tools.md`: no secrets, no credentials, no real customer/tenant data in what you hand it. Scope what it can see. (For multi-tenant specifics, see `references/data-protection.md` and `references/my-environment.md`.)
- **The handoff-bundle → Claude Code path is "packaging" — review it like agent-authored code.** Generated front-end is AGENT-AUTHORED by definition: read **every** diff, ride the **same** CI and the **same** a11y gates from §3, and confirm **WCAG AA + working light/dark + responsive** (against SKILL.md's breakpoints) before merge. No exception for "the tool made it."
- **"Or better" means stay tool-agnostic.** The standard is the **OUTPUT** — tokens, AA conformance, light/dark, responsive, beautiful — not the producer. Claude Design, a Figma-plus-handoff workflow, or hand-coded by you all clear the same bar or don't ship. Tool choice is a convenience; the gate is fixed.
