# Multi-Agent & Shared-Repo Coordination

Companion reference for the senior-engineering-partner skill. This is the CONCURRENCY OVERRIDE to the Definition of Done: it takes precedence the moment a second writer is in the tree.

When more than one agent (or a human plus an agent) works the same codebase concurrently, the solo-speed defaults below become hazards. **An agent is a developer** — so the discipline here is simply professional team practice (GitHub Flow / trunk-based development with branch protection), applied to autonomous workers. Treat this whole section as the **concurrency override** to the **Definition of Done** above: where that section optimizes a lone developer for speed ("commit each change, push immediately to `main`"), the rules below take precedence the moment a second writer is in the tree.

## Isolation is the default — one workspace, one branch, one task per agent
Never let two agents write the same working tree at the same time. Give each its own checkout and integrate through review, not through a shared filesystem.
- **One working tree per agent.** Use `git worktree add ../task-x -b feat/task-x` (cheap, shares the object store) or a separate clone. Each agent edits only inside its own tree.
- **One branch per task, never commit straight to `main`.** Protected `main` is the integration point, not a scratchpad. Each task is a short-lived feature branch (`feat/…`, `fix/…`) that lands via PR.
- **Single-writer ownership for un-branchable state.** Some state can't be branched and merged — a live database, a running daemon's runtime files, an external system's config. Assign exactly one agent as its writer for the duration; others read or queue requests. Concurrency control is ownership, not locking after the fact.
- **Ownership transfers explicitly, never implicitly.** When the writer of un-branchable state hands off (task done, or another agent needs it), it's an explicit step: the outgoing owner leaves the state at a known-good checkpoint (committed/applied, not mid-edit) and says so; the incoming owner confirms receipt before writing. "I'll assume nobody else is touching it" is exactly the race ownership exists to prevent.
- **Don't touch a file another agent is editing.** Before starting, establish who owns which paths. Overlapping edits to the same file are the most common collision — partition by file/module, not by line.

## Integrate through PR + CI, not through racing pushes
This is the override of the Definition of Done's "push immediately" rule. Under concurrency, the integration gate is the PR, and the CI on it is what makes parallel work safe.
- **Branch protection on `main` (required).** Require PR review and passing status checks before merge; disallow direct pushes and force-pushes to `main`. This is what makes "an agent is a developer" safe — the gate is mechanical, not trust-based.
- **Required CI as the merge gate.** The same automated QA this skill already mandates — tests, `bandit`/`npm audit`, ShellCheck, linters — runs in CI on every PR and **blocks merge on failure**. Locally-green is necessary but not sufficient; the PR's checks are the source of truth. Wire these as required status checks so no PR (human or agent) merges red.
- **`git pull --rebase` before every push.** Rebase your branch onto the latest `origin/main` (or onto the base branch) before pushing or opening the PR, so you integrate others' landed work and resolve conflicts in *your* tree, not in the shared one. Linear history; never `git pull` with an implicit merge in a multi-writer repo.
- **On a rebase conflict: resolve in your own tree, or bail and surface it — never guess.** A conflict means another writer changed the same lines. If it's in code you own and understand, resolve it locally, re-run the gates, then push. If it's in a shared module, a migration sequence, or code you don't own, **stop and surface it** rather than force-resolving — a wrong resolution silently reintroduces a bug or drops someone's landed change. Never `git checkout --ours/--theirs` blind across a file to make a conflict "go away."
- **Never `git add -A` / `git add .` in a shared tree.** Stage by explicit path (`git add path/to/file`). A blanket add in a tree another agent is also writing sweeps up their in-flight edits into your commit — the same "flag, don't absorb, stray changes" rule from the Definition of Done, made stricter because here the stray changes are *active*, not leftover.
- **Small, frequent PRs over one big branch.** Short-lived branches reduce conflict surface and let CI catch problems early. A long-running agent branch that diverges for hours is a merge-conflict generator.
- **Structured PR description, every time.** Reuse the **What changed / Why it changed / Testing instructions** format from Source Code Management so a reviewing agent or human has full context without reading the diff cold.

## Right tool for the job — don't do collaborative development in a sync repo
A single-writer **sync** mechanism is not a collaboration platform. Any "apply the source over the target" tool assumes one authoritative writer reconciling state — it has no merge/review model, and its automation actively fights concurrent editors.
- **Develop in a real Git repo; sync only the finished artifact.** If two agents need to build something together, do it in a proper repo with branches and PRs, then publish/sync the result. Do not treat a single-writer sync source tree as a shared dev branch — an auto-apply job and an auto-sync `git add . && git commit` will clobber or absorb in-flight work (see **Machine-synced config** in SKILL.md; that discipline is the single-writer assumption this rule generalizes).
- **Match the concurrency model to the tool.** Code → branch + PR + CI. Synced config → single writer, commit promptly, let it apply. Runtime state → one owner. Picking the wrong model is the root cause of nearly every multi-agent collision.

## Before you start, after you finish
- **Open with a coordination check:** confirm your branch/worktree, confirm which paths you own, `git fetch && git rebase origin/main` so you start from current state.
- **Close with the Definition of Done, gated by the PR:** working tree clean, branch pushed, PR opened with required CI green, no files touched outside your owned paths. "Done" in a multi-agent context means *merged via a green PR*, not *pushed to a branch*.

## Subagent-driven development (one author delegating — distinct from the concurrency override above)
The sections above govern *multiple independent writers* racing in a tree. This is the opposite shape and worth using deliberately (it's the technique Anthropic's Claude Code guidance and obra/superpowers both push): **one orchestrator decomposes the work and dispatches subagents with isolated context**, staying in control of integration.
- **Separate research/plan from execution.** Do exploration in **plan mode** (or a read-only research pass) and agree the spec/plan *before* touching code — the `engineering-workflow.md` spec→plan gate, applied to agent orchestration. This stops "solved the wrong problem with great rigor."
- **Dispatch implementer subagents with fresh, scoped context.** For a large task, hand each unit to a subagent that gets only the files/spec it needs — this keeps the main context clean and each unit focused. Useful for read-heavy investigation or a well-bounded slice; it is *not* a license to skip review.
- **Use a separate reviewer pass.** Have a distinct subagent (or your own `REVIEW:` mode) check the implementer's diff against the spec and the security/isolation dimensions before it lands — the author and the reviewer shouldn't be the same context. This is the same "human reviews every agent PR / don't self-merge blind" rule (`github-teams.md`), applied within a session.
- **Still gate on the real Definition of Done.** Subagent output is agent-authored code: it goes through the same branch→PR→required-CI path and the verify-before-asserting checks. Delegation changes *who drafts*, never *what gates*.
